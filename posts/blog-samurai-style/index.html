Linting legacy monoliths

2022-01-12 — Written by Greg

Alright, who committed styling changes on 13000 files to main on the day before we need to merge 5 long-lived feature branches for a release cycle?

- A Legacy Developer

The Problem

Introducing code standards to large legacy repositories can seem straight forward, but there are several shortcomings that must be accepted or addressed.

Too large

Some issues that can I’ve come across are that some code bases too large for regular scanning. They overwhelm the linting tool and take several minutes to an hour to complete. That isn’t practical when you want enforce standards on every repository push.

Disruption downstream

Other downsides are successfully running the tool and then using an auto-fix feature to update the code base at once. This en masse update sounds like a great plan, but if these updates are pushed downstream to multiple long-lived feature branches, it can become a nightmare to sort out by requiring delicate merge strategies.

Regression

This type of action may introduce unwarranted risk to the product and force a full regression test. Quite a few legacy systems aren’t fully automated, so this could be considered unreasonable and rejected by product owners/managers.

Buy-in

Code quality is a concept that’s difficult for people who don’t have their hands in the code everyday to recognize how important it is. It’s especially critical in larger applications where comments, indentation and general readability are needed for developers to be comfortable and confident.

So what can be done?

Quickest result is to just pull the band-aid off, push the styling changes en masse and deal with the repercussions. Results could consist of angry developers due to complex merges, delayed QA tasks, regression testing, and –perish the thought– a root cause analysis meeting.

Another option is for gradual introduction of coding conventions. Reformat the code only when it’s being touched will lead to eventual consistency.

A problem with this method is that without tooling it’s a manual process that would require discipline and trust in your teammates. Code reviews would help out, but code style should not be the focal point of a code review. There is a need for automated tools.

Gradual integration via tooling

Since I’m primarily a Java developer, this tool focuses around that ecosystem. There may be similar tools for other frameworks/languages that can be used interchangeably.

This section details gradual coding conventions implemented with Samurai Style Cop, a tool for java projects using checkstyle. Similar methods can be used for your tool of choice.

Checkstyle is an excellent tool because it can be incorporated into the build process in multiple ways: cli, maven/gradle plugin, ant task, etc.

Whatever tool you use, determine if it supports suppressions. Checkstyle is very convenient because not only can it suppress files, but has enough granularity to suppress lines within a file. So if the incoming change is only for line 50 through 57, it will ignore warnings for other lines.

Use the git diff

So to produce a suppression file, let’s start with getting a list of changed files. We can do that by running a git diff against our local repository.

git --no-pager diff --minimal -U0 -- *.java

The above example uses --no-pager so that the output is dumped unimpeded to the screen without using an interface like less. Remove if you want to scroll.

Additionally, --minimal is used so create the smallest possible diff and -U0 turns off context lines.

Brief overview of git diff format (2-way, patch)

Git Diff is an extension of the Unified Diff Format with a few changes to the header.

All patch headers start with a diff --git header line. a/ and b/ is the filename before and after the changes. Renames will have a different b/ filename. If the file is new or removed, the filenames will be identical. The status of the file will be described later in the header, if applicable.


diff --git a/src/Main.java b/src/Main.java

Some diffs will have extended information. In the above information, there is an index header. This is a common extended header that displays the shortened blob object names and the full file mode (<hash>..<hash> <mode>). Blob object names are from my understanding the name of the file’s versioned content stored in the git database. The mode portion is a record of the file type and permissions bits.


index 8dc6c1e..43c108b 100644

Other extended headers can also be present: 


old mode <mode>
new mode <mode>
deleted file mode <mode>
new file mode <mode>
copy from <path>
copy to <path>
rename from <path>
rename to <path>
similarity index <number>
dissimilarity index <number>

The rest of the diff is Unified format.

--- represents the filename prior to the changes. +++ is the filename after changes were made. This should be verbatim of what is in the header with the exception of the possibility of seeing /dev/null for either line when the file is created or deleted. The diff --git header line does not utilize /dev/null.


diff --git a/examples/pixel-aspect-ratio/Cargo.toml b/examples/pixel-aspect-ratio/Cargo.toml
new file mode 100644
index 0000000..aaf4732
--- /dev/null
+++ b/examples/pixel-aspect-ratio/Cargo.toml

Following the original/modified filenames is the diff hunk header. This header starts and ends with 2 ampersands and is defined in the following format:

@@ -<Line Number Start>,<Length> +<Line Number Start>,<Length> @@

The minus - and plus + corresponds with the original and modified file revision respectively. Line number starting refers to the line number in the file where the differences start and the length is the total number of lines that the change consists of. So if the change starts on 14 and ends on line 16, the value would be 14,3. The range is inclusive of the starting line (14, 15, 16 would be three lines.)

If no number is after the line start, such as in the example of @@ -13,0 +16 @@, the range is 1.

A range of 0 indicates that no lines were modified, and the change is distinctly an addition or removal.

The next part of the diff hunk are the content lines. Each content line starts at column 1, rather than column 0. Column 0 is reserved for - or +, which indicates if the line was removed or added.

In this example hunk, lines 25 and 26 were removed in the old revision of the file, and then starting on line 28 of the new revision, lines 28, 29, 30, and 31 were added. The lines may or may not match up depending on the number of changes in a prior hunk.

Multiple hunks can be defined per file diff, as well as multiple file diffs may be present in a patch.

Enter the Samurai

Samurai Style Cop has a simple parser for git diff files using the above definition. It tracks the line numbers and file paths for every file mentioned in the diff, then builds a Checkstyle suppressions.xml file. This file can be used with an external or embedded Checkstyle executable. If using embedded, it only passes the paths to Checkstyle to scan which significantly reduces the amount of files to scan for most changes.

Since we only scan the diff and not the individual files, there are caveats.

The length of the file (the max num of lines) is just a static guess of 50,000. The number can be tweaked in the command line (--limit switch), but it’s a constant used for every file in the diff. Checkstyle doesn’t care if the file is shorter than the max line, but files that are larger would have lines beyond that scanned.

As an example, a scanned file is 127 lines long. We want to suppress all but lines 3, 4, and 5. The suppression file would look something like this:

<suppress checks=".*" files="Main.java" lines="1-2,6-50000"/>

Since it tracks the file path, the program could be extended to scan each file for length prior to producing the suppression markup.

Areas of Interest in the Code

  • Diff Scanner/Parser
    • Features a state machine for more accurate transitions and stop false positive state parsing
    • Simple parsing using Regular Expressions
    • Not a true parsing library, only implements enough of the git diff format to process the suppression list.
  • Embedded Checkstyle
    • Suppression list builder and setup for embedded Checkstyle.

Integrate into your build process

Currently, client-side pre-commit git hooks are the only method that I personally looked into. It puts the onus on the developer to setup and not deactivate, so this might not be ideal for all users.

I could imagine setting up Jenkins to run the Checkstyle process and either failing the build or setting it as unstable would be relatively simple. I plan on setting up a Jenkins container and testing this out in the future. I will post an update when this occurs.
